# Disclaimer

**Prospect Assist AI is a proof-of-concept built for the IDBI Innovate 2026 hackathon (Problem Statement 2). It is a prototype, not a production banking system, and nothing in this repository or its live demo should be treated as a real IDBI Bank product, service, or decision.**

---

## 1. All customer/lead data is synthetic

Every customer record used by this application is generated by `backend/scripts/generate_data.py`. There is no real customer data anywhere in this codebase, in the demo, or in any exported file.

Concretely, the generator (seeded with `np.random.seed(42)` for reproducibility) fabricates 20,000 fictitious customers with:

- **Identity/demographics**: name (via the `Faker` library's `en_IN` locale), age, gender, city, occupation, employer, employment type — none map to a real person.
- **Account and income data**: account tenure, average monthly balance, salary/other-income credits — drawn from statistical distributions (normal, lognormal, Poisson), not sourced from any bank system.
- **UPI spend-category signals**: 15 categories (groceries, rent, EMI payments, SIPs, entertainment, etc.), each sampled from an exponential distribution — not real transaction data.
- **Salary-credit velocity / spend-discipline signals**: a synthetic day-of-month salary credit, percentage of income spent within 3 days, and days-to-balance-depletion, derived arithmetically from the fabricated income and spend figures above — not observed account behavior.
- **Digital behavior**: loan-page visits, calculator usage, time on loan pages, last-visit-days-ago — randomly sampled, not real clickstream.
- **Credit bureau fields**: credit score, existing loan count, EMI burden, utilization, payment history — randomly generated, not pulled from any bureau.
- **Life-event flags**: salary hike, new rent, education spend increase, medical spike, marriage indicators, investment maturity, vehicle insurance lapse — random binary flags with fixed probabilities, not detected from any real event.
- **Outcome label** (`actually_took_loan`): a synthetic binary label produced from a hand-tuned probability formula so the intent model has something to learn from — it does not reflect any real loan booked by any real customer.

No production IDBI Bank system, database, or API is read from or written to at any point.

## 2. Scored outputs are illustrative model outputs, not real decisions

The Intent, Capacity, and Propensity scores, the composite lead score, and the Hot/Warm/Cold (RAG) status shown for every lead are computed by the rules-and-model pipeline in `backend/scoring/models.py` and `backend/scoring/train.py`, running entirely on the synthetic data described above.

- These scores are **illustrative outputs of a prototype scoring pipeline**, not real credit decisions, not real suitability assessments, and not a substitute for IDBI Bank's actual credit/underwriting process.
- The "capacity" score's need/want/retained-income segmentation and the "spend-discipline" score are derived from fabricated UPI-category and salary-timing data, not from an actual bank-verified cash-flow analysis.
- The "data-quality/confidence" flag reflects how many *synthetic* data sources are populated for a given fabricated record — it does not reflect the real-world reliability of any actual customer's data.
- The intent model's reported accuracy/AUC figures (see `backend/data/model_performance.json`) describe how well an XGBoost classifier fits a synthetic, self-generated label on synthetic features. They say nothing about real-world loan conversion.

No output from this application should be used, directly or indirectly, to approve, deny, price, or otherwise act on any real credit or financial product for any real person.

## 3. No real customer PII

Because all data is synthetically generated (Section 1), there is no real personally identifiable information (PII) anywhere in this repository, its data files, or its demo — no real names, phone numbers, account numbers, addresses, or identity documents. Any resemblance between a generated name and a real person is coincidental (`Faker` samples from a generic name pool).

## 4. AI-drafted RM pitch scripts are starting points, not ready-to-send content

The `POST /recommend/{customer_id}` endpoint in `backend/app/main.py` (function `generate_recommendation`) calls the DeepSeek LLM API to draft a short phone-call script and supporting talking points for a relationship manager (RM), personalized with the fabricated lead's name, occupation, and recommended product. If no DeepSeek API key is configured, or the call fails for any reason, the endpoint falls back to a fixed template string instead.

Whether AI-generated or template-generated, this text is:

- A **draft starting point** for an RM, not a compliance-approved script.
- **Not fact-checked** against the customer's actual real-world circumstances (because there is no real customer behind it in this prototype).
- **Not reviewed** for regulatory, suitability, or fair-lending language.

**A human RM must review, verify, and, if necessary, edit any AI-drafted script before it is used in any real conversation with any real customer.** This application does not send anything to a customer on its own — it only drafts text for a human to review.

## Known Limitations

These are honest, current limitations of the prototype as it exists in this repository — not aspirational or hidden gaps.

| Area | Limitation |
|---|---|
| **Data** | 100% synthetic; no real bank data source (core banking, UPI switch, credit bureau) is integrated. Scores and segmentation are only as realistic as the generator's distributional assumptions. |
| **Intent model** | A single XGBoost classifier trained on 5 behavioral features against a synthetic conversion label. Reported AUC/accuracy in `model_performance.json` reflect fit to synthetic data, not a validated real-world model. |
| **Capacity / need-want-retained segmentation** | Computed entirely from synthetic UPI-category totals; there is no verification that a real customer's UPI feed would map cleanly onto these 15 categories, and no bureau cross-check for accounts a customer may hold at other institutions. |
| **LLM-drafted pitches (DeepSeek)** | Output is not guaranteed factually accurate, regulatorily compliant, or free of hallucination. Falls back to a generic template on any API failure — the fallback is also not customer-specific beyond name/product. |
| **Prompt-input sanitizer** (`backend/scoring/safety.py`) | The guardrail added in front of the DeepSeek call is a **heuristic, regex- and length-based check** on the three fields (`name`, `occupation`, recommended product) interpolated into the LLM prompt. It is not an exhaustive prompt-injection defense: it catches known patterns (instruction-override phrasing, persona-switch phrasing, oversized strings) but a sufficiently novel phrasing could evade it. In this prototype the practical exposure is low because these fields come from the synthetic dataset via a `customer_id` lookup, not from raw free-text user input — but the sanitizer exists on the defense-in-depth principle that any value entering an LLM prompt should be checked, regardless of its nominal source. |
| **No authentication** | The API has no auth layer; `CORSMiddleware` is configured with `allow_origins=["*"]`. Not suitable for exposure beyond a hackathon demo. |
| **No confidence calibration validation** | The `confidence_level`/`data_completeness_score` fields count how many synthetic data sources are populated; they are not validated against any real-world notion of data trustworthiness. |
| **No audit trail** | Unlike some governance patterns, this prototype does not persist a record of what was blocked/sanitized by the guardrail or what was sent to the LLM — it only logs to stdout at request time. |
| **Test coverage** | The pytest suite (`backend/tests/`) covers the scoring functions' boundary behavior and the input sanitizer. It does not cover the FastAPI endpoints end-to-end, the frontend, or the DeepSeek integration itself (no network calls are made in tests). |
